### `assets/gallery-data.json`

Current references:

- `gallery.html` through `data-gallery-source="assets/gallery-data.json"`
- `zh/gallery.html` through `data-gallery-source="../assets/gallery-data.json"`
- `tools/generate_gallery_data.py` as its output

- `.github/workflows/deploy-pages.yml` indirectly, because the workflow runs `tools/generate_gallery_data.py`

No JavaScript currently reads the `data-gallery-source` attribute. Therefore, this file is still part of the generation/extraction toolchain, but it is not the manifest used by the current page renderer.

### `assets/data/gallery.json`

Current references:

- Generated from the Excel `gallery` sheet by `tools/generate_site_data.py`
- Loaded directly by `scripts/data-renderer.js`
- Used by both English and Chinese Gallery pages through their shared renderer

### Actual runtime behavior

The Gallery pages actually render `assets/data/gallery.json`. The legacy `data-gallery-source` attributes remain in the HTML, but the active JavaScript does not consume them.

### Excel-driven coverage

The Excel-driven workflow does cover Gallery content:

```text
data/website_content.xlsx, sheet gallery
→ tools/generate_site_data.py
→ assets/data/gallery.json
→ scripts/data-renderer.js
→ English and Chinese Gallery pages
```

However, the deployment workflow also regenerates `assets/gallery-data.json` from the album folder before generating the Excel-based JSON. It does not copy newly discovered gallery rows back into Excel. As a result, adding an image to `assets/albums/` updates the legacy manifest but does not automatically add that image to the actual runtime `assets/data/gallery.json` unless the Excel gallery sheet is also updated.

### Recommended future unification

The preferred single runtime source is `assets/data/gallery.json`, because it is consistent with the rest of the Excel-driven website.

Before unification, a dedicated Gallery phase should decide how new album files enter the Excel-driven data:

1. Make album scanning update or merge the Excel gallery sheet before `generate_site_data.py`; or
2. Make `generate_site_data.py` derive Gallery rows directly from the album folder while preserving any Excel metadata; or
3. Keep Excel as the sole source and explicitly document that new images require adding a gallery row to Excel.

Likely files involved in a future unification:

- `gallery.html`
- `zh/gallery.html`
- `scripts/data-renderer.js` only if the runtime contract changes
- `tools/generate_gallery_data.py`

## Naming and organization findings

- Directory casing is mixed by category: `Project` and `CV` use uppercase names, while `albums`, `albums-optimized`, and `data` use lowercase names.
- Current references match that casing, so this is an organization issue rather than a runtime defect.
- GitHub Pages is case-sensitive. Case-only renames on Windows are high risk and require a deliberate Git rename procedure.
- Spaces and non-ASCII characters are common in image filenames. Current encoded paths work, but path changes require URL encoding and production verification.
- JPG/WebP pairs in `assets/Project/` and the source/optimized Gallery directories are intentional pipeline pairs, not safe duplicate-removal candidates.
- Profile and SEO images are mixed with other asset categories: the portrait is at the root of `assets/`, while the shared SEO image is also a project source image.

## Risks

- Moving a project WebP without updating Excel can cause regenerated JSON to restore the old broken path.
- Updating Excel without regenerating JSON can leave the deployed site inconsistent with the workbook.
- Moving Gallery images before resolving the dual flow can break the Excel path, generated JSON, legacy JSON, and optimization pipeline differently.
- Case-only directory renames may appear successful on Windows but fail after deployment to Linux-based GitHub Pages.
- Renaming files containing spaces or non-ASCII characters can produce mismatched encoded and decoded URLs.
- Moving CV PDFs can break both hard-coded HTML downloads and Excel-generated link data.
- Moving the shared high-throughput JPG can break Open Graph and Twitter previews across all public pages.
- Removing source JPG files because WebP versions exist would break future optimization/regeneration.
- Moving `assets/data/` requires coordinated renderer and deployment changes across every data-driven page.

## Recommended next action

Do not move any asset immediately.

The next phase should first unify the Gallery ingestion/data flow, because it is the only area where the build-time source and runtime source can diverge after a normal content update. After that is resolved and validated, perform a small manual review of the four unreferenced `Longlong_Li_CV_*` files. Profile/SEO image normalization should come later as a coordinated, low-volume path migration.
